## Вопрос 1 — Zone redundancy requirement
Вопрос:
What is the requirement for enabling zone redundancy in your Container Apps environment?

Варианты ответов:
- The environment must include a virtual network (VNET) with an available subnet.
- The environment must be deployed in a single availability zone.
- The environment must not use a virtual network (no VNET required).
- The environment must be created using Azure Kubernetes Service (AKS).

Правильный ответ:
- The environment must include a virtual network (VNET) with an available subnet.

Подробное объяснение:

1) Зачем VNET и подсеть для zone redundancy — концепция

 - Размещение по зонам (Availability Zones) подразумевает, что платформа сможет создать один или несколько экземпляров ваших сетевых компонентов/инстансов в физически разнесённых дата‑центрах внутри региона. Чтобы эти экземпляры корректно общались с внешней и внутренней сетью и чтобы у них были предсказуемые IP‑адреса и маршруты, требуется управляемая сетевая среда — VNET.
 - Подсеть внутри VNET даёт конкретный диапазон IP (CIDR) и границы, в которых Azure создаёт сетевые интерфейсы для инстансов. Когда включена зональная отказоустойчивость, платформа может развернуть дополнительные сетевые интерфейсы/инстансы в других зонах, и они должны получить адреса в той же подсети, иначе сеть и маршрутизация нарушатся.

2) Технические требования и делегирование

 - Для Container Apps (и некоторых других управляемых сервисов) платформа требует делегирования подсети выбранному сервису (например, делегирование подсети сервису контейнеров — Microsoft.App или эквивалент). Делегирование позволяет сервису управлять ресурсами в этой подсети (создавать интерфейсы, Nat/PIP, маршруты и т.д.).
 - Подсеть должна иметь достаточный пул свободных адресов (IP) для размещения всех экземпляров в каждой зоне и для будущего масштабирования.

3) Почему без VNET/подсети платформа не может обеспечить размещение по зонам

 - Если среда не привязана к пользовательскому VNET, платформа использует свои управляемые сети, где Azure не гарантирует размещение отдельных экземпляров в вашем контролируемом адресном пространстве и не предоставит необходимого делегирования или маршрутизации. В результате платформа не сможет корректно распределять ресурсы по зонам и управлять адресацией.

4) Практические рекомендации

 - Выделите подсеть с запасом IP (в зависимости от ожидаемого числа инстансов и сервисов). Часто рекомендуют подсети /24 или больше для продуктивных сред, но выбор зависит от ваших потребностей.
 - Убедитесь, что подсеть делегирована сервису Container Apps (или соответствующему сервису), и что NSG/маршруты не блокируют необходимые соединения.

## Вопрос 2 — Managed identities в scaling rules
Вопрос:
You want to use managed identities in the scaling rules for your container app. Which ones can you use?


Варианты:
- System-assigned identities only
- User-assigned identities only
- Both
- None

Правильный ответ:
- None

Подробное объяснение:

1) Как работает автоскейлинг в контейнерных платформах (ключевая идея)

 - В Azure Container Apps автоскейлинг выполняется платформным контроллером (в основе часто используется KEDA — Kubernetes-based Event Driven Autoscaler или собственные сервисные контроллеры Azure). Этот контроллер читает метрики/события/правила и инициирует создание/удаление инстансов.
 - Контроллер действует от имени платформы, а не от имени конкретного контейнера/ресурса пользователя.

2) Что такое managed identity и почему её нельзя применить внутри правила скейлинга

 - Managed identity (system‑assigned или user‑assigned) выдается и ассоциируется с конкретным Azure‑ресурсом (например, с контейнерным приложением, виртуальной машиной или функцией). Токены и разрешения, выдаваемые этой identity, доступны в контексте этого ресурса.
 - Контроллер автоскейлинга, который читает метрики и принимает решения, не выполняется «в контексте» вашего ресурса, поэтому он не может использовать токен вашей managed identity, привязанной к приложению. Проще говоря — identity принадлежит приложению, а не самому контроллеру, который совершает вызовы при скейлинге.

3) Последствия и альтернативы

 - Из‑за этого нельзя напрямую указать в правилах скейлинга, что платформа должна использовать system‑assigned или user‑assigned identity приложения для доступа к внешнему ресурсу при вычислении метрик.
 - Варианты обхода:
	 - Хранить секреты/строки подключения в секретах Container Apps (или в Key Vault) и использовать их в триггерах/скейлерах, если скейлер поддерживает работу с такими значениями.
	 - Использовать внешние скейлеры или промежуточные компоненты (например, Azure Function, Logic App или отдельный сервис), которые могут аутентифицироваться (имея свою managed identity или service principal) и отправлять метрики в систему скейлинга.
	 - Настроить скейлинг по метрикам, не требующим доступа к защищённым ресурсам (CPU, память, HTTP‑запросы, очередь сообщений и т.д.).

4) Пример паттерна

 - Если вам нужно скейлить по длине очереди в стороннем хранилище, можно сделать так: отдельный сервис (Function) с собственной managed identity читает длину очереди и публикует кастомную метрику в мониторинг (Application Insights/Prometheus/Custom Metrics), а контроллер автоскейлинга использует эту метрику для масштабирования.

## Вопрос 3 — Команда PowerShell для Azure Container Instances
Вопрос:
You are tasked with deploying a .NET Core application on Azure, and you plan to use Azure Container Instances for this purpose. Which PowerShell command should you use to create the necessary resources for hosting this application?



Варианты:
- New-AzContainerService
- New-AzContainerGroup
- First New-AzContainerGroup, then New-AzContainerService
- Either New-AzContainerGroup or New-AzContainerService
- None of the mentioned

Правильный ответ:
- New-AzContainerGroup

Подробное объяснение и пример

1) Разница между ACI и Container Service/AKS

 - Azure Container Instances (ACI) — это максимально простой способ запустить контейнеры в Azure без управления кластером. ACI организуются в контейнерные группы (container groups), которые содержат один или несколько контейнеров, совместно использующих сеть и, при необходимости, общий том.
 - Azure Container Service / AKS — это управление кластерами Kubernetes; команды/ресурсы, относящиеся к Container Service, относятся к кластеризованным решениям и не подходят для простого одно‑ или многоконтейнерного развёртывания в ACI.

2) Команда PowerShell

 - Для создания контейнерной группы используется `New-AzContainerGroup`.
 - Пример базовой команды (PowerShell):

```
New-AzContainerGroup -ResourceGroupName "rg-demo" -Name "myContainerGroup" -Image "mcr.microsoft.com/dotnet/samples:aspnetapp" -OsType Linux -IpAddressType Public -Port 80 -Cpu 1 -MemoryInGb 1.5 -RestartPolicy Always
```

 - Этот вызов создаёт контейнерную группу с указанным образом, публичным IP и портом. Для реального приложения вы, возможно, захотите добавить переменные окружения, привязку томов, настройки сети (VNET), секреты и т.д.

3) Когда использовать `New-AzContainerService`

 - `New-AzContainerService` применяется для создания более сложных управляющих ресурсов и исторически ассоциируется с сервисами оркестрации; для простых развёртываний ACI он не нужен.

4) Дополнительно

 - Если нужно развернуть .NET Core приложение в контейнере как ACI для тестирования или простого хостинга — используйте `New-AzContainerGroup`.
 - Если нужна оркестрация, persistent volumes, сложный сетевой контроль или долгоживущие рабочие нагрузки — рассмотрите AKS и соответствующие команды.
